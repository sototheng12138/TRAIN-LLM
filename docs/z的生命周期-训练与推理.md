# z 的生命周期：训练阶段挨打进化，推理阶段执行裁决

本文档说明辅助头 logit **z** 在训练与推理两阶段的角色，以及为何在论文中必须强调“训练阶段”。

---

## 1. 训练阶段：z 用来“挨打”并“进化”

- **过程**：输入训练集数据 → 前向得到 **z**（二分类 logits）→ 通过 Softmax 得到置信度 C（P(有发运)）。
- **监督**：此时有伪标签 **c_disp**（1=有货，0=没货，由未来窗口是否发运生成）。
- **惩罚（Loss）**：BCE Loss。若真实为 0 而 z 给出高置信度，Loss 大，反向传播“惩罚”模型。
- **进化**：更新 W_aux、b_aux 及上游表示。目标：**没货时 z 尽量小，有货时 z 尽量大**，从而学会区分“发运前兆”与“空闲前兆”。

**对应脚本**：`scripts/Iron.sh`（带 `--use_aux_loss` 的训练）。

---

## 2. 推理阶段：z 用来“执行裁决”

- **过程**：输入完全未见过的历史数据 → 前向得到 **z**（无标签，仅凭训练好的权重）。
- **裁决**：由 z 得到 C = P(有发运)。若 **C < 0.5**，系统判定：数值头在该窗口的“小值/毛刺”为幻觉。
- **动作**：对该窗口的数值预测执行**整窗置 0**，并输出【系统决策说明】供调度员直接采用。

**对应脚本**：`scripts/Eval_Iron_AuxDecision.sh`（带 `--output_aux_semantic` 与 `--aux_confidence_threshold 0.5`）。

---

## 3. 为什么必须在论文里强调“训练阶段”？

审稿人容易担心：**清零规则是否在推理阶段“拍脑袋”定的？**

需要写清：

- **虽然清零动作发生在推理阶段，但执行清零的依据（z 的判断能力）是在训练阶段，通过大量真实业务样本（2023–2025 数据）与 BCE 严格训练得到的。**
- 推理阶段的 0.5 阈值只是对“有发运/无发运”二分类的决策边界，**z 的尺度与判别力来自训练**，而非人工调参。

---

## 4. 总结：z 的生命周期

| 阶段     | z 的角色           | 结果 |
|----------|--------------------|------|
| 训练阶段 | 通过 Loss 训练 z 的准确性 | 学会分辨发运前兆与空闲前兆 |
| 推理阶段 | 用学好的 z 生成置信度 C   | C 作为开关，决定是否抹平数值头毛刺 |
| 结果输出 | —                  | 置信度 + 系统给 0 + 调度员可执行决策 |

**一键推理裁决**（需先完成训练）：

```bash
bash scripts/Eval_Iron_AuxDecision.sh
```

**完整流程**（训练 + 推理裁决）：

```bash
bash scripts/Iron.sh
bash scripts/Eval_Iron_AuxDecision.sh
```
