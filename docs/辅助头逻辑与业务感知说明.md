# 辅助头逻辑与「业务感知」说明

## 一、辅助头出现在哪些地方

| 环节 | 位置 | 作用 |
|------|------|------|
| **训练** | `run_main.py`：前向时 `return_aux_repr=True`，得到 `aux_logits`；与伪标签算 `L_aux`（CE），与主损失相加 | 联合损失 L_total = L_main + λ·L_aux，梯度回传 |
| **推理/评估** | `run_eval.py`：前向时 `return_aux_repr=True`，取 `aux_logits` 做 softmax → 0/1 + 置信度，再与数值统计（有发运天数）一起填模板 | 输出「是否有发运」及置信度、天数，不参与损失 |

所以：**辅助头既参与损失（训练），也参与输出（推理）**；不是“只有在损失函数有”，推理时也会用同一头给出发运判定。

---

## 二、辅助头在模型中的位置与梯度

- **前向**：  
  `x_enc` → 归一化 → patch → 重编程层 → 与 prompt 拼接 → **冻结 LLM** → `dec_out` → 取最后若干 patch 的隐状态 `block` → **repr_pooled = block.mean(dim=(1,3))** → **aux_has_shipment(repr_pooled)** → aux_logits。
- **repr_pooled**：来自 **LLM 的 last hidden state** 对 patch 维的池化，即「LLM 对当前时序输入编码后的聚合表示」。
- **训练时梯度**：  
  L_aux 对 aux_logits 求导 → 对 repr_pooled 求导 → 对 block、dec_out 求导 → **穿过冻结的 LLM**（只反传、不更新 LLM 参数）→ 对送入 LLM 的 `enc_out`（重编程层输出）求导 → 对 **重编程层、patch 嵌入** 求导。  
  因此：**LLM 参数不更新，但「重编程层 + patch 嵌入」会收到 L_aux 的梯度**，被鼓励产生「经 LLM 编码后既能预测数值，又能区分有/无发运」的表示。

---

## 三、能否达到「让 LLM 底层特征学会业务感知」？

- **若把「业务感知」理解为**：  
  模型在提取特征时，会下意识地编码与「是否有发运」相关的信息（发运信号），而不是只拟合数值。  
  **可以达到**，但需要把表述说准。

- **更准确的说法**：  
  - **LLM 本身不更新**，其“底层特征”的权重不变。  
  - 被训练的是：**送入 LLM 的时序表示**（patch + 重编程层输出）。在 L_main + λ·L_aux 的联合训练下，这些可训练部分会学习生成一种表示，使得：  
    - 经冻结 LLM 编码后的 **repr_pooled** 既能被数值头预测好（主任务），又能被辅助头区分「有发运 / 无发运」（辅助任务）。  
  - 因此，**学到“发运信号”的是骨干里的可训练部分（重编程层、patch 嵌入）**，它们会下意识地往「对发运判别有利」的方向编码时序；LLM 相当于固定的“编码器”，其输出被我们用来同时做数值预测和发运分类。

- **论文/说明里可以这样写**：  
  「辅助损失使骨干中可训练部分（重编程层、patch 嵌入）在联合训练下学会生成兼具数值预测与发运判别能力的表示，从而在特征中编码“发运信号”，实现业务感知；LLM 保持冻结，不更新其参数。」

---

## 四、小结

- 辅助头：**训练时参与损失 L_aux，推理时参与输出**（发运 0/1、置信度、与数值统计一起填模板）。  
- 梯度从 L_aux 经辅助头回传到 **repr_pooled → LLM（仅反传）→ 重编程层、patch**，所以能促进“发运信号”被编码进**送入 LLM 的表示**里。  
- 「让 LLM 底层特征学会业务感知」若指「让整体表征具备业务感知」，**能达到**；若指「让 LLM 内部权重学会」，则不对——应改为「让骨干中可训练部分学会生成具有业务感知的表示，经冻结 LLM 编码后同时服务数值预测与发运判定」。
