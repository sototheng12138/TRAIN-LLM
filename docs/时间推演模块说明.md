# 时间推演模块：流程说明与「事件推演」定位

## 一、过程说明（具体流程）

### 1. 触发条件与数据准备（run_eval.py）

- **何时执行**：跑评估时，若测试集具备 `data_x`、`inverse_transform`，且预测结果 `pred` 非空，则进入时间推演分支。
- **取哪条样本**：只对**第一个 batch 的第一条样本**做语义生成（首窗口）。
- **数据来源**：
  - **历史**：`hist_orig = test_set.inverse_transform(test_set.data_x[:seq_len])`，形状 `(seq_len, n_channels)`，原始量纲。
  - **预测**：`pred_first = pred[0]`，形状 `(pred_len, n_channels)`，已在前面逆变换到原始量纲。

### 2. 语义生成的三段式结构

整体输出由三块**顺序拼接**而成：

| 段落 | 来源 | 输入 | 生成方式 |
|------|------|------|----------|
| **【趋势】+【发运连续性】+【品种关系】+【预测期总体】** | `utils/railway_semantic.py` | `hist_orig` + `pred_first` + `col_names` | 纯规则：对数值做统计与阈值判断，再套固定模板 |
| **【重编程层文本原型】**（可选） | 同上 `generate_prototype_reasoning` | 首 batch 的 patch→原型 attention + 原型词表 | 对 attention 权重排序，取 top-k 原型并查词，套模板 |
| **【辅助任务推断结果】**（可选） | run_eval 内联 | 首 batch 的辅助分类头 logits + `pred_first` | 二分类「是否有发运」+ 置信度；结合数值统计出「有发运天数」，套模板 |

下面只展开**第一段**（铁路业务语义）的规则逻辑，因为这是你看到的「趋势 / 发运连续性 / 预测期总体」的直接来源。

### 3. 铁路业务语义的规则逻辑（railway_semantic.py）

- **【趋势】**  
  - 对每个通道（品种）：  
    - 历史近期 = 历史序列末尾与预测等长的一段，求均值 `mean_hist`。  
    - 预测期 = 整段预测求均值 `mean_pred`。  
  - 规则：  
    - 历史≈0、预测>0 → “由无发运转为有发运，需关注车皮与装卸安排”。  
    - 相对变化 ≥10% → “上升约 X%” 或 “下降约 X%”。  
    - 否则 → “与历史近期水平相当，走势平稳”。

- **【发运连续性】**  
  - 对每个通道：在预测期按**时间步**算「接近零」的占比（`|x|<1e-6`）。  
  - 规则：  
    - 零占比 ≥50% → “超过一半日期无发运，呈批次/专列模式”。  
    - ≥20% → “部分日期无发运，建议按日核对计划与车皮”。  
    - 否则 → “发运较连续，可优先保障该品种排产稳定性”。

- **【品种关系】**（若存在）  
  - 用预测期各通道「是否有发运」（如 >1e4 为有）做二值序列，两两算相关系数。  
  - 高正相关 → “常同时有发运，联合排产或共线发运”；高负相关 → “此有彼无，运力竞争或计划交替”。

- **【预测期总体】**  
  - 预测期各通道求和 → 总发运量，格式化为「约 X 万吨」。  
  - 各通道占比排序，取主导品种（占比 ≥20% 的列出来）→ “以 XX、XX 为主，建议预留车皮与装卸能力”。

以上**全部是确定性规则 + 数值统计 + 中文模板**，无大模型、无学习、无外挂脚本。

### 4. 输出写入

- 三段拼接后的整段 `semantic` 会：  
  - 在终端打印；  
  - 写入 `ckpt_dir/semantic_output.txt`（带标题「基于首窗口：历史 X 天 + 预测 Y 天」）。  
- 若启用 `--output_aux_semantic`，第三段会包含辅助分类头的「是否有发运」及「至少 X 天」的表述。

## 二、是否可以作为「事件推演模块」

### 当前定位：**时间维度的数值推演 + 模板化业务解读**

- **“推演”体现在**：  
  - 输入是**历史 + 预测**两段数值序列；  
  - 在**时间轴**上做统计（均值、零占比、相关性），得到「趋势」「连续性」「品种关系」「总量与结构」；  
  - 输出是**与时间/发运相关的事件化描述**（如“由无发运转为有发运”“部分日期无发运”“以 XX 为主”）。

- **“事件”的粒度**：  
  - 不是“第 3 天发生 A、第 7 天发生 B”的细粒度事件，而是**窗口级、通道级、统计级**的结论，例如：  
    - 某品种在预测期内“由无到有”“上升/下降约 X%”“连续/不连续”；  
    - 品种间“同时发运/此有彼无”；  
    - 预测期总体“约 X 万吨、以 XX 为主”。  
  - 辅助任务再补一层**窗口级二分类**（是否有发运）和**有发运天数**（由数值统计得到）。

因此，**可以把它称为「事件推演模块」**，但需要明确其含义是：

- **事件** = 由数值规则归纳出的**业务状态/结论**（趋势、连续性、主导品种、是否有发运、发运天数等），而不是细粒度时间点事件。  
- **推演** = 在给定「历史 + 预测」数值序列上，用**固定规则 + 模板**做统计与解释，**不做额外学习、不调用大模型**。

### 若希望更贴近“事件推演”的直觉

可在现有框架上做**扩展**（不改变现有逻辑），例如：

1. **按日/按区间输出“事件”**  
   - 对 `pred_first` 按日或按周做阈值判断（如“有发运日”“显著放量日”），输出「第 t 天：品种 X 有发运」之类的列表，再交给模板或简单 NLG 成句。  
2. **事件类型更细化**  
   - 在现有趋势/连续性/品种关系之外，增加规则（如“某日多品种同时起量”“某品种连续 N 天为 0 后首次非零”），对应更多模板句。  
3. **与外部事件对齐（若有标注）**  
   - 若有“检修日”“节假日”等标签，可在模块内增加“预测期内包含哪些已知事件”的说明，与数值结论并列输出。

这样，**当前实现已经是一个可用的、基于规则的事件推演模块**；若要强调“事件”感，只需在文档/对外表述中明确「事件 = 由数值推演得到的业务结论」，并在需要时按上面方式做扩展即可。
